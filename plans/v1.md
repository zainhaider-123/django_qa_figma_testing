# QA Figma Testing Tool — v1 Plan

## Overview
A Django web app that connects to a user's Figma account (OAuth 2.0 via django-allauth), lets them pick a project → page → frame, enter a live site URL, and runs a pixel-perfect comparison between the Figma design render and a Playwright screenshot of the live site. Results are shown in a visual report with mismatch score, side-by-side, and diff overlay.

## Key Architectural Decision: Figma OAuth + REST API (NOT MCP)
Research confirms the Figma Dev Mode MCP server is built for LLM/agent tool-calling inside IDEs — it has no HTTP interface for a Django backend, no project-listing tools, and operates on a single node given a Figma URL. It is unsuitable for a multi-user web app that browses projects and exports frames. **v1 uses Figma OAuth 2.0 + REST API via django-allauth.** MCP can be revisited later as an alternative input path.

| DB | SQLite (existing) | Fine for v1; swap to Postgres later |

## Figma API Endpoints Used
| Purpose | Endpoint | Scope |
|---|---|---|
| Current user + team_id | `GET /v1/me` | `current_user:read` |
| List team projects | `GET /v1/teams/{team_id}/projects` | `projects:read` |
| List files in project | `GET /v1/projects/{project_id}/files` | `projects:read` |
| File tree (pages + frames) | `GET /v1/files/{file_key}?depth=2` | `file_content:read` |
| Export frame as PNG | `GET /v1/images/{file_key}?ids={node_id}&scale=2&format=png` | `file_content:read` |

### OAuth 2.0 Flow (via django-allauth)
django-allauth handles the standard OAuth2 Authorization Code Grant flow:
1. User clicks "Connect Figma" → allauth redirects to Figma authorize URL with scopes: `file_content:read projects:read current_user:read`
2. Figma redirects back with auth code → allauth exchanges for access_token + refresh_token
3. Tokens stored in `SocialToken` / `SocialAccount` models (allauth-managed)
4. Access token: 90-day lifetime. Refresh via allauth's token refresh mechanism or custom refresh logic.
5. Custom Figma OAuth2 adapter registered with allauth (FigmaProvider) to define authorize/access/refresh URLs and scopes.

## User Flow (Step by Step)

### 1. Connect Figma Account
- User clicks "Connect Figma" → django-allauth OAuth flow → SocialToken stored
- If token expired, auto-refresh using allauth's refresh mechanism

### 2. Select Project
- `GET /qa/new/` → fetches user's teams → projects → renders project list
- User selects a project → shows files in that project

### 3. Select Page/Frame
- User selects a file → fetch `GET /v1/files/{file_key}?depth=2`
- Render tree: pages (CANVAS) → frames (FRAME) with thumbnails
- User selects a frame → captures `node_id`, `frame_width`, `frame_height`

### 4. Enter Site URL + Run
- Form: site URL + optional threshold override
- On submit → create `TestRun` (status=pending) → dispatch Celery task → redirect to run status page

### 5. Celery Task (`qa.tasks.run_comparison`)
1. Fetch Figma frame PNG: `GET /v1/images/{file_key}?ids={node_id}&scale=2&format=png` → download to `figma_image`
2. Screenshot live site: Playwright sync API, viewport = `{width: frame_width*2, height: frame_height*2}` (match scale=2), `device_scale_factor=1`, `wait_until="networkidle"` → save to `site_screenshot`
3. Normalize: ensure both images are same pixel dimensions (resize screenshot to match figma image if needed)
4. Compare: `pixelmatch(img1, img2, diff_img, threshold=run.threshold)` → get `mismatch_pixels`
5. Compute: `mismatch_pct = (mismatch_pixels / (w * h)) * 100`
6. Determine pass/fail: `pass` if `mismatch_pct <= threshold_pct` (default threshold: 1.0%)
7. Save all images + metrics to TestRun → status=completed (or failed on exception)

### 6. View Report
- `GET /qa/run/{id}/` → report page showing:
  - Pass/Fail badge + mismatch percentage (large)
  - Three images side-by-side: Figma design | Live screenshot | Diff overlay
  - Toggle: slider overlay (drag to reveal design vs screenshot)
  - Run metadata: URL, frame name, dimensions, timestamp, threshold
  - "Re-run" and "New Test" buttons

## Implementation Phases

### Phase 1 — Foundation & Figma OAuth (django-allauth)
- Add dependencies to requirements.txt
- Install + configure django-allauth in settings.py (AUTHENTICATION_BACKENDS, INSTALLED_APPS, SOCIALACCOUNT_PROVIDERS)
- Create custom Figma OAuth2 provider adapter (FigmaProvider) defining:
  - `authorize_url = https://www.figma.com/oauth`
  - `access_token_url = https://api.figma.com/v1/oauth/token`
  - `refresh_token_url = https://api.figma.com/v1/oauth/refresh`
  - `profile_url = https://api.figma.com/v1/me`
  - scopes: `file_content:read projects:read current_user:read`
- Configure SocialApp with Figma client_id + secret (via admin or env)
- Create `figma` app with `FigmaClient` service class that retrieves the user's SocialToken and makes authenticated REST API calls
- Settings: `FIGMA_CLIENT_ID`, `FIGMA_CLIENT_SECRET`, `FIGMA_REDIRECT_URI` via decouple
- Deliverable: user can connect Figma account via allauth, token stored + refreshed

### Phase 2 — Project & Frame Selection
- `FigmaClient` methods: `get_me()`, `get_team_projects()`, `get_project_files()`, `get_file_tree()`, `get_frame_image()`
- Views + templates: project list → file list → frame tree (pages → frames)
- Frame selection stores node_id + dimensions in session/form
- Deliverable: user can browse and select a frame

### Phase 3 — Test Run + Comparison Engine
- Create `qa` app, `TestRun` model, migrations
- Set up Celery + Redis (`celery.py`, `settings.py` config)
- `ScreenshotService`: Playwright sync wrapper (viewport from frame dims, networkidle wait)
- `CompareService`: pixelmatch wrapper (returns mismatch count + diff image)
- `qa.tasks.run_comparison` Celery task orchestrating: fetch → screenshot → compare → save
- View: create run form (URL + threshold) → dispatch task → status page
- Deliverable: user can submit a URL and a comparison runs in background

### Phase 4 — Report UI
- Report view + template: pass/fail badge, mismatch %, side-by-side images, diff overlay
- Image slider toggle (CSS/JS) for design vs screenshot comparison
- Run history list (`GET /qa/runs/`)
- Re-run button (creates new TestRun with same params)
- Deliverable: complete report viewing experience

### Phase 5 — Polish & Hardening
- Error handling: Figma API errors, Playwright timeouts, image dimension mismatches
- Token expiry auto-refresh (allauth + custom refresh logic)
- Rate limit awareness (Figma: 300 req/min)
- Playwright browser install step documented (`playwright install chromium`)
- Login required on all views
- .env.example updated with all new vars
- Deliverable: robust v1 ready for use

## Risks & Mitigations
| Risk | Mitigation |
|---|---|
| Figma image export URL expires (30 days) | Download + store PNG in TestRun immediately |
| Live site dimensions don't match frame | Resize screenshot to match figma image dims before compare |
| Playwright not installed | Document `playwright install chromium` in setup |
| Token expires mid-session | allauth token refresh + custom refresh decorator on FigmaClient calls |
| Slow sites cause timeout | Playwright `timeout` config + retry once |
| Anti-aliasing false positives | pixelmatch handles AA by default (configurable) |
